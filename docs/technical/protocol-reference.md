# LCC/OpenLCB Protocol Reference

*This document provides curated technical reference for LCC protocol implementation. It supplements (but does not replace) the authoritative standards in `markdown/standards/`. All claims cite source specifications.*

**Primary Sources:**
- TN-9.7.x series (Technical Notes) in `markdown/standards/`
- OpenLCB protocol specifications
- Reference implementations

## Protocol Overview

### Producer/Consumer Event Model

**Core Concept:** LCC uses a publish-subscribe event model where producers generate events and consumers respond to them, without direct knowledge of each other.

**Key Characteristics:**
- **Decoupled:** Producers don't know which consumers exist
- **Many-to-Many:** One event can have multiple producers and consumers
- **Event IDs:** 8-byte unique identifiers (64 bits)
- **Asynchronous:** Events flow through the network without acknowledgment

**Example:**
```
Producer                Event Network               Consumer
--------                -------------               --------
Button Press  →  Event: 05.02.01.02.00.00.00.03  →  Light ON
                       (broadcast to all nodes)
```

**Source:** TN-9.7.3.1 (Event Transport)

### Event ID Format

**Structure:** 8 bytes (64 bits), displayed as dotted hex notation

```
05.02.01.02.00.00.00.03
│  │  │  │  └─────────┘
│  │  │  │      └─ Unique portion (often sequential)
│  │  │  └──────── Node-specific
│  └──┴────────────── Manufacturer/range assigned portion
└───────────────────── Often relates to node ID prefix
```

**Display Formats:**
- **Human:** `05.02.01.02.00.00.00.03` (dotted hex)
- **Wire:** 8 raw bytes `[0x05, 0x02, 0x01, 0x02, 0x00, 0x00, 0x00, 0x03]`
- **Internal:** Often stored as hex string `"0502010200000003"`

**Important:** Event IDs are generated by nodes, never by applications. Applications can copy/move Event IDs between elements but should not invent new ones.

**Source:** TN-9.7.0.4 (Event Identifiers)

### Typical Event ID Patterns

**Producer Elements:**
- Usually have 1 Event ID per state (e.g., "Event On", "Event Off")
- Simple one-to-one relationship with physical action
- Example: Button press → produces Event A

**Consumer Elements:**
- Often have multiple Event ID slots (Event 1-6 typical)
- Allows single consumer to respond to multiple different events
- Example: Light can turn on from button 1 OR button 2 OR timer 1

**Duplicate Event IDs by Design:**
- Multiple producers can share the same Event ID (intentional)
- Multiple consumers can listen for the same Event ID (intentional)
- This is NOT an error condition but core to the protocol

**Source:** TN-9.7.3.1 §3

## Node Discovery Protocol

### Verify Node ID

**Purpose:** Discover all nodes on the network

**Protocol Flow:**

1. **Send Global Query:**
   ```
   MTI: 0x19490 (Verify Node ID Global)
   Source: Our node alias
   Data: None (or specific Node ID to query)
   ```

2. **Nodes Respond:**
   ```
   MTI: 0x0170 (Verified Node ID)
   Source: Responding node alias
   Data: 6-byte Node ID
   ```

3. **Silence Period:**
   - After sending query, wait for timeout period (typically 250-500ms)
   - Collect all responses during this window
   - Silence indicates discovery complete

**Implementation Notes:**
- Use global broadcast (no specific destination)
- Responses arrive asynchronously
- Nodes may respond out of order
- Duplicate responses possible (filter by Node ID)

**Source:** TN-9.7.3 §4.2 (Message Network Standard)

### Node ID and Alias

**Node ID (Unique Identifier):**
- 6 bytes (48 bits)
- Globally unique across all LCC networks
- Format: `XX.XX.XX.XX.XX.XX` (dotted hex)
- Example: `05.02.01.02.FF.01`
- Permanent for the life of the node

**Alias (Temporary Address):**
- 12 bits (0x000 - 0xFFF)
- Dynamically allocated at startup
- Used for frame addressing (saves bandwidth)
- Can change across reboots
- Format: `0xXXX` (3-digit hex)
- Example: `0x3A5`

**Relationship:**
- Alias is short-hand for Node ID during a session
- Alias resolution happens via Verify Node ID protocol
- Lost alias requires re-discovery

**Source:** TN-9.7.0.3 (Unique Identifiers)

## SNIP Protocol

### Simple Node Ident Protocol

**Purpose:** Retrieve human-readable node information (manufacturer, model, versions, user-assigned names)

**Protocol Flow:**

1. **Send SNIP Request:**
   ```
   MTI: 0x19DE8 (SNIP Request)
   Destination: Target node alias
   Data: None
   ```

2. **Node Responds with Datagram:**
   ```
   MTI: 0x1A08 (Datagram - varies by frame position)
   Source: Target node alias
   Data: SNIP payload (multi-frame if large)
   ```

3. **Send Datagram ACK:**
   ```
   MTI: 0x19A28 (Datagram ACK OK)
   Destination: Target node alias
   ```

**SNIP Data Structure:**

```
Version (1 byte) = 4
Manufacturer Name (null-terminated string)
Model Name (null-terminated string)
Hardware Version (null-terminated string)
Software Version (null-terminated string)
User-Provided Node Name (null-terminated string)
User-Provided Description (null-terminated string)
```

**Parsing:**
- First byte is version (currently 4)
- Six null-terminated strings follow
- Strings may be empty (just null byte)
- Maximum SNIP payload ~254 bytes (datagram limit)

**Common Issues:**
- Some nodes don't support SNIP (respond with error or silence)
- Timeout required (500ms typical)
- Multi-frame datagrams need reassembly

**Source:** TN-9.7.4.3 (Simple Node Information Protocol)

## Datagram Protocol

### Multi-Frame Message Assembly

**Purpose:** Send messages larger than 8 bytes (GridConnect frame data limit)

**Frame Types:**

```
MTI: 0x1A## - Datagram frames
  0x1A28 - First frame only (complete in one message)
  0x1A48 - First frame of multi-frame
  0x1A68 - Middle frame
  0x1A88 - Final frame
```

**Assembly Process:**

1. Receive first frame (0x1A48) → Start buffer
2. Receive middle frames (0x1A68) → Append to buffer
3. Receive final frame (0x1A88) → Complete datagram
4. Send ACK (0x19A28) or NAK (0x19A48)

**Buffer Management:**
- Maximum datagram size: typically 254 bytes
- Timeout per fragment (discard incomplete datagrams after ~1s)
- Track by source alias (multiple simultaneous datagrams possible)

**Error Handling:**
- NAK indicates rejection or error
- Retry logic in sender (not receiver)
- Dropped frames detected by timeout

**Source:** TN-9.7.3.2 (Datagram Transport)

## GridConnect Frame Format

### ASCII Frame Structure

**Format:** `:X[HEADER]N[DATA];`

**Components:**
- `:` - Start delimiter
- `X` - Extended frame marker
- `[HEADER]` - 8 hex digits (MTI + addressing)
- `N` - Data separator
- `[DATA]` - 0-16 hex digits (payload bytes)
- `;` - End delimiter

**Example:**
```
:X195B4123N0102030405060708;
  │  │ │└─ Destination alias (0x123)
  │  │ └── Source alias (0x4?? embedded)
  │  └──── MTI portion (0x195B)
  └─────── Frame type indicator
  
Data: [01, 02, 03, 04, 05, 06, 07, 08] (8 bytes)
```

**Header Decoding:**
- Bits layout varies by MTI and addressing mode
- Alias encoding uses clever bit-packing
- See TN-9.7.2.1 for complete bit-field layout

**Data Encoding:**
- Each byte becomes 2 hex digits
- 0-8 bytes typical for most messages
- Empty data section valid (e.g., `:X0490N;`)

**Important:** Frame parsing must be strict. Invalid frames should be logged but not crash the application.

**Source:** TN-9.7.2.1 (CAN Frame Transfer)

**Categories:**

| MTI Range | Purpose |
|-----------|---------|
| 0x04xx | General Messages (Verify Node ID, etc.) |
| 0x05xx | Producer/Consumer Identified |
| 0x09xx | Event Messages |

**Addressed vs Global:**
- Global: Broadcast to all nodes
- Addressed: Sent to specific node alias

**Source:** TN-9.7.3 (Message Network Standard)

## CDI (Configuration Description Information)

### Purpose

CDI is an XML document that describes a node's configuration structure. It defines what can be configured, data types, constraints, and user-visible names.

**Key Concepts:**

**Segments:** Major sections (e.g., "Port I/O", "Conditionals")  
**Groups:** Collections within segments (e.g., "Lines 1-16")  
**Elements:** Individual configuration items (e.g., "Line 3 Event On")  
**Replication:** Groups can repeat (e.g., 16 lines, each with same structure)  

**CDI Structure (Example):**

```xml
<cdi>
  <segment space="253">
    <name>Port I/O</name>
    <group replication="16">
      <name>Line</name>
      <repname>Line</repname>
      <eventid>
        <name>Event On</name>
      </eventid>
      <eventid>
        <name>Event Off</name>
      </eventid>
      <int size="1">
        <name>Debounce (ms)</name>
        <min>0</min>
        <max>255</max>
      </int>
    </group>
  </segment>
</cdi>
```

**Retrieval:**
- CDI stored in memory space 0xFF (read-only)
- Retrieved via Memory Configuration protocol
- Typically 10-100KB (can be slow)

**Caching Strategy:**
- Cache key: `{manufacturer}_{model}_{software_version}`
- Reuse across nodes with same firmware
- Store in application data directory

**Source:** TN-9.7.4.1 (Configuration Description Information)

## Configuration Memory Operations

### Memory Spaces

| Space | Purpose | Access |
|-------|---------|--------|
| 0xFF | CDI XML | Read-only |
| 0xFD | Configuration | Read/Write |
| 0xFE | All Memory | Read/Write (deprecated) |
| 0xFC | Firmware Upgrade | Write-only (special) |

**Configuration Read/Write:**
- Address-based: Read/write specific byte ranges
- Protocol: Memory Configuration Protocol with datagrams
- Atomic: Changes take effect when written (no "commit" needed)

**Best Practice:**
- Read current value before write (verify before modify)
- Write only changed values (not entire configuration)
- Confirm write with read-back

**Source:** TN-9.7.4.2 (Memory Configuration Protocol)

## Event Discovery

**Status:** ✅ Implemented in Feature 006 (Bowties Event Discovery) — addressed variant used

### Identify Events Protocol

**Purpose:** Discover which events a node produces or consumes

**Global Query:**
```
MTI: 0x0997 (Identify Events Global)
Data: None
```

**Addressed Query (used in Bowties feature):**
```
MTI: 0x0968 (Identify Events Addressed)
Data: Destination node alias (2 bytes)
```

Bowties uses the addressed form — one message per known node, 125 ms apart — to limit network traffic and ensure replies are attributable to a specific node.

**Responses:**
```
MTI: 0x0544 (Producer Identified Valid)
Data: 8-byte Event ID

MTI: 0x04C4 (Consumer Identified Valid)
Data: 8-byte Event ID
```

**States:**
- **Valid:** Event is configured and active
- **Invalid:** Event slot exists but not configured
- **Unknown:** Event state cannot be determined

**Note:** The `EventState` field in replies (Valid/Invalid/Unknown) is currently ignored by `query_event_roles`; only the MTI itself (Producer vs Consumer) is used to assign roles.

**Use Case:**
- Map all event relationships on network
- Build event link (bowtie) diagrams
- Detect orphaned events (producer with no consumer)

**Source:** TN-9.7.3.1 §5 (Event Transport)

## Protocol Best Practices

### Timing and Timeouts

**Discovery Timeout:** 250-500ms (balance speed vs completeness)  
**SNIP Timeout:** 500-1000ms (datagram reassembly takes time)  
**Datagram Fragment Timeout:** 1000ms (discard incomplete)  
**CDI Retrieval:** Can take 5-30 seconds (large XML, slow protocol)  

### Concurrency Control

**SNIP Queries:**
- Limit concurrent requests (5-10 max)
- Prevents network congestion
- Improves reliability on slow hubs

**Discovery:**
- Single global query, collect all responses
- Don't query nodes individually (inefficient)

### Error Handling

**Network Errors:**
- Retry with exponential backoff
- Log errors but don't crash
- Provide user-friendly error messages

**Protocol Violations:**
- Log malformed frames
- Continue processing other messages
- Report to user if persistent

**Graceful Degradation:**
- SNIP not supported? Use Node ID as name
- CDI retrieval failed? Show raw memory view
- Timeout? Allow user to retry with longer duration
