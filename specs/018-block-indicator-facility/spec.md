# Feature Specification: Block Indicator Facility — Channels, LED Indicators, and the First Facility

**Feature Branch**: `018-block-indicator-facility`
**Created**: 2026-06-27
**Status**: Draft
**Input**: Introduce a layered pin → channel → facility model in which channels are the only first-class binding entity — every channel has a **role** (what it does — e.g., reporting block occupancy, indicating a lamp state) and a **style** (the specific hardware shape that realises that role — e.g., a BOD-8 input on a daughter board, a single LED driven from Direct Lamp Control). Channels come in two ownership flavours: **hardware-owned** (auto-created and auto-deleted by hardware-config choices, e.g., selecting a BOD daughter board) and **user-owned** (created and deleted by the user through facility-slot actions). Facilities may have empty slots — that is how an incomplete plan is represented in this slice. Add a `lamp-indicator` role and a `single-led-direct-lamp` style for Signal LCC Direct Lamp Control. Deliver a Block Indicator behavior template. Together: a user can scaffold a facility, fill its input slot by selecting one of the auto-created BOD channels, fill its output slot by adding a new lamp-indicator channel bound at creation time to a specific lamp row, and watch the facility become Wired the moment both slots are filled. Shipped with release notes and user documentation that note the feature is early and subject to change.

## Context

Information channels with live state already work end-to-end for block-occupancy (specs 015/016/017). What does not yet exist is:

- **A way to act on information.** No consumer channels, no output side of the I/O picture.
- **A way to compose channels into named behavior.** No concept of a "facility" that ties producer channels to consumer channels through a named purpose.
- **A way to scaffold a layout incrementally.** Channels today are coupled to live hardware and the facility concept does not yet exist; the user cannot lay out the named-behavior shape of their layout (facilities + slots) and bind channels into it as nodes come online.

This feature delivers all three by centring the data model on a single first-class entity — the **channel** — that always carries its own **role** (what the channel does in the layout) and **style** (the specific hardware shape that realises that role). The Block Indicator behavior itself is a trivial pass-through (`occupied → lit`, `clear → unlit`); the point of choosing this slice is to land the channel-and-facility architectural seams on a problem whose end state is verifiable with a physical LED, while keeping intermediate states (facility with empty slots, facility partially filled) demoable and testable.

The feature ships visible to all users so feedback informs its evolution. Expectation-setting is handled out-of-product through two lightweight channels: the **release notes** for the version that ships this feature (explicitly describing what's new, what's likely to change, and how to give feedback), and the **user documentation** under `docs/user/` (a short section introducing facilities, marked as evolving). The Facilities section itself ships as plain functional UI with no in-product badges, banners, or expectation-setting chrome — the intent is to invite real use and feedback while keeping the UI calm.

### The three-layer model (architectural decisions already made)

The data model separates three concerns:

| Layer | Knows about | Requires hardware? | Examples |
|---|---|---|---|
| **Pin** | Physical I/O location on a node | Yes | A BOD-8 input on a connector; a Direct Lamp Control row on a Signal LCC |
| **Channel** | Identity, name, role, style, binding (pin(s) on a node), ownership | Yes — every channel is always bound | `block-occupancy` channel "Block 5" with style `bod-block-detector-input` bound to BOD-8 input 3; `lamp-indicator` channel "Block 5 Indicator" with style `single-led-direct-lamp` bound to Direct Lamp Control row 7 |
| **Facility** | Behavior template + slots each with an **optional** channel reference | No — empty slots are valid | Block Indicator named "Block 5" with both slots filled (Wired); or with one or both slots empty (Incomplete) |

A channel always has a non-null role, a non-null style, and a non-null binding; every channel is tied to specific hardware. Facilities reference channels (not pins). Each layer has its own lifecycle: a facility can be renamed or deleted independently of its slot channels; a slot's channel can be unbound from the slot (slot becomes empty) without deleting the facility.

**Channel role vs channel style.** Every channel is described by two orthogonal facts:

- The **role** is what the channel does in the layout — its state vocabulary plus the slot-binding contract. Examples: `block-occupancy` (states `unknown`/`occupied`/`clear`); `lamp-indicator` (states `unknown`/`lit`/`unlit`); future `signal-aspect-3-color` (states `unknown`/`stop`/`approach`/`clear`). A facility slot binds **by role** — the input slot of a Block Indicator requires a `block-occupancy` channel, regardless of which style implements it.
- The **style** is the specific hardware shape that realises a role — pins claimed, event-leaf mapping, CDI-field constraints. Examples: `bod-block-detector-input` (1 input pin on a BOD daughter board, realises `block-occupancy`); `single-led-direct-lamp` (1 Direct Lamp Control row, realises `lamp-indicator`); future `3-led-direct-aspect` and `2-led-bicolor-aspect` (both realise `signal-aspect-3-color` but on different pin counts and wiring shapes).

A role may be realised by one style (this slice has 1:1 for both roles) or by several styles in the future. The user-facing language is "role" and "style"; the formal data-layer mechanism (one style matches one role's state vocabulary so slot-binding works) is an implementation detail described under Functional Requirements.

**Ownership decides how channels come into existence and disappear.** Two flavours of channel coexist in this slice:

- **Hardware-owned channels** are auto-created the moment a hardware-configuration choice fixes the role of pins — selecting a BOD-family daughter board on a connector says "these pins are block-occupancy detectors, full stop, they can't do anything else while this board is selected." Bowties responds by creating one channel per configured input pin with role `block-occupancy` and style `bod-block-detector-input`, with an auto-generated default name. The user can rename these channels for layout meaning ("Mainline-block-3"), but rename does **not** change ownership — when the daughter board is cleared or changed, all of its hardware-owned channels disappear, including renamed ones. Any facility slot bound to a deleted channel becomes empty; the facility transitions back to Incomplete. Names are lost on hardware-config changes, which is acceptable because daughter-board changes are infrequent and intentional.
- **User-owned channels** are created on demand by user action and persist until the user removes them. A user-owned channel is created via a facility slot's **Add channel** action; the action picks the style (when more than one is available for the slot's role) and the binding target (e.g., a specific Direct Lamp Control row) as part of channel creation, because there is no logical state in this slice for a channel to live in. Removing a user-owned channel from its only slot deletes it (no ref-counting in this slice — that lands later when channels are shared across slots).

**Empty slots are how this slice represents "planned but not yet wired."** A user can create a Block Indicator facility, leave both slots empty, and save the layout — the facility's scaffold persists and round-trips. As the user configures hardware and binds channels into the slots, the facility advances toward Wired. Pre-creating unbound channels is intentionally **not** supported in this slice; full hardware-less planning requires a placeholder-node mechanism that is tracked under Future Considerations.

**Role matching.** A facility slot declares the role it requires; any channel of that role can fill the slot, regardless of its style. Styles that realise richer hardware — for example a multi-LED signal head — would realise a different, domain-specific role (something like `signal-aspect-3-color`, whose states are signalling-domain values such as `clear`/`approach`/`stop`), not `lamp-indicator`, because their state vocabulary is fundamentally different from a single lit/unlit lamp. For this feature there is exactly one style per role, so role-matching can stay implicit — but role and style remain separate concepts from day one.

**Styles claim a list of node-side endpoints** (not a single one). For this feature an endpoint is always a physical pin — a BOD input on a connector, a Direct Lamp Control row — and the rest of this spec uses "pin" as shorthand. The list shape is plural from day one so multi-endpoint styles (e.g., a three-aspect signal claiming three lamp rows) can land later without a data refactor, and the "endpoint" framing is intentionally broader than "pin" because future producer or consumer sources may not be physical I/O at all (e.g., the output of an on-node Logic block, or any other addressable unit a node exposes). Generalising the term is out of scope here; the seam is shaped to permit it.

### Roles carry domain-meaningful state values

A role IS the state-vocabulary contract, and that vocabulary names real-world intent rather than electrical or boolean abstractions. Every role includes `unknown` as a first-class state for "no observation yet" — used while a channel is bound but the bus has not yet delivered a relevant event.

The two roles this feature introduces:

- **`block-occupancy`** — states `unknown`, `occupied`, `clear`. Not `true`/`false`, not `on`/`off`. The block is occupied or clear; that is what the layout cares about.
- **`lamp-indicator`** — states `unknown`, `lit`, `unlit`. Not `on`/`off` (a generic electrical abstraction). The lamp is lit or unlit. ("On"/"off" describes a switch; "lit"/"unlit" describes a lamp.)

Embedding intent at the role boundary keeps the rest of the stack honest. A future `turnout-position` role would expose `unknown`/`normal`/`reverse`, never `on`/`off`, even if its underlying style drives a single binary output.

### Facility lifecycle: Incomplete → Wired

A facility is born from a behavior template with all its slots empty. As the user fills slots, the facility's status advances:

- **Incomplete** — at least one slot is empty. No underlying bowtie(s) exist for the template's mapping. The facility exists in the layout (and round-trips on save/reopen) but no producer-to-consumer wiring has been laid down.
- **Wired** — all slots are filled by bound channels and the underlying bowtie(s) that realise the template's mapping exist (created using Bowties' existing bowtie creation mechanism). Producer events drive consumer events through the bus as the template prescribes; live state of bound channels updates as the bus reports events. Whether the wiring edits have reached every node yet is owned by the existing layered storage system (in-memory / on-disk / on-bus) and is not part of facility status.

Status transitions are automatic: the facility becomes Wired the moment the last slot becomes filled, and returns to Incomplete the moment any slot loses its channel (channel removed from slot, channel disappears because its underlying hardware-config was cleared, facility deleted). There is no separate "deploy" action and no intermediate status in this slice. Becoming Wired creates the underlying bowtie(s) using the existing bowtie creation mechanism; returning to Incomplete frees them via the existing slot-detach pipeline. Spec 018 introduces no new sync, persistence, or deployment machinery — facilities are a UI veneer that drives bowtie creation. (A future feature could split off an explicit deploy step if staging changes ever becomes a need.)

**Facility persistence stores the declaration, not just the resulting event wiring.** Slot bindings + template id + channel identities are persisted so the facility can be reconstructed from declaration alone — that keeps future YAML-template loaders on the same code path as today's hardcoded templates.

### Styles impose configuration constraints

A style declares a **constraint contract** over the CDI fields of the pins its channels claim. When a channel exists on a pin, the pin's CDI fields fall into two categories:

- **Managed fields** — the style fixes the field to a specific value, restricts it to a subset of allowed values, or hides it entirely. The user cannot put the field into a state that would invalidate the channel's semantics.
- **Unmanaged fields** — the style has no opinion; the field stays freely user-editable.

Constraints exist for the entire life of a channel — from the moment it is created (hardware-owned channels: when the daughter board is selected; user-owned channels: when Add channel completes) until the moment it is destroyed.

This is the same mechanism that already governs daughter-board selection today (e.g., BOD-8's `validityRules` in `RR-CirKits.shared-daughterboards.yaml`). This feature **repositions that mechanism as a property of the style, not of the selection event**:

- BOD-8 retrofit: the constraints currently under the daughter-board entry move to the `bod-block-detector-input` style. User-visible behavior is identical — selecting BOD-8 still applies the same field restrictions to the same fields — but the constraints now travel with the style that owns the channel claiming the pin.
- Lamp-indicator on Direct Lamp Control: the `single-led-direct-lamp` style declares its own constraint contract. At minimum it manages `Lamp Selection` (must not be "Used by Mast", which would mean the lamp is controlled by mast aspect logic instead of direct command).

**Basic enforcement** (fixing values, restricting allowed sets, hiding) is in scope. **Constraint UX polish** (collapsed-by-default managed sections, drift detection from external edits, one-click repair) is deferred.

### Block-occupancy retrofit

The pre-018 block-occupancy channel shape (channel directly holding `connector` + `input`, with no separate style layer) is replaced by the new channel-with-role-and-style model. After the retrofit, **daughter-board selection creates one hardware-owned channel per configured input pin**, each with role `block-occupancy` and style `bod-block-detector-input` bound to its pin, with a default name the user can rename. The producer event-leaf mapping (occupied/clear) lives on the style's definition, not in channel-creation code. No parallel persistence shape is acceptable.

**Daughter-board-bound channel lifecycle.** Selecting a BOD-family daughter board on a connector is a hardware-capability decision: it fixes those pins to be occupancy-detection inputs, period — the hardware cannot do anything else while that daughter board is selected. Because the meaning of those pins is determined by the board choice, the hardware-owned channels for those pins appear when the board is selected and disappear when the selection is cleared or changed. Any facility slot bound to a deleted channel becomes empty; the facility transitions back to Incomplete. User-owned channels (e.g., lamp-indicator channels on Direct Lamp Control rows) follow the user-driven lifecycle described elsewhere in this spec.

**Hardware verification via the Channels panel.** The Railroad tab introduces a **Channels** section: a hardware-organised list of every channel in the layout, showing its location (node + subsystem + pin), its role, its style, its live state, and the slot/facility currently bound to it (or "unbound"). This is the surface for the "is my hardware wired correctly and working" question — connecting to the bus and triggering a real BOD-8 input lights up the corresponding channel row immediately, with no facility required, because the BOD daughter board has already produced channels with bindings the moment it was selected. The Channels panel also makes the slot's Select-channel picker obvious: when you click Select on a facility's input slot, you see the same hardware-owned channels you've been watching in the panel, filtered to those of the slot's required role and not currently bound to another slot.

**Producer vs. consumer asymmetry on the Channels panel.** Producer channels (those with the `block-occupancy` role, currently always with style `bod-block-detector-input`) display genuinely observed bus state. Consumer channels (those with the `lamp-indicator` role, currently always with style `single-led-direct-lamp`) display the most recently observed command on the bus for the row's event ids — there is no physical-state feedback from Direct Lamp Control hardware, so this is "last commanded," not "last confirmed." Users need a way to discover this distinction (so they do not mistake commanded state for physical confirmation), but it does not need to be surfaced as an inline label on every consumer-channel row — a tooltip, the release notes, and the user documentation together are sufficient. Sending a test command to verify a consumer channel is deferred (see Future Considerations).

**Hardware claim state surfaces through existing profile constraints, not new Config-tab UI.** When a lamp-indicator channel claims a Direct Lamp Control row, the style's constraint contract (FR-026–FR-030) becomes active on that row's CDI fields — the user sees `Lamp Selection` locked to its style-required value, while unmanaged fields stay editable. This is the same profile-driven validity/relevance mechanism that already governs daughter-board selections; this feature adds no dedicated Config-tab claim-state badges or per-row "claimed by [facility]" annotations. Users who want a complete claim-state view get it from the Channels panel (FR-031), whose binding column already names the slot/facility for every channel.

**No migration of pre-018 layouts.** Single user, pre-1.0; the user manages their own pre-018 data and ensures only post-018 layouts are opened with this version. No migration code, no detection logic, no compatibility surface. The spec 015 per-input channel inventory is retired in the final implementation slice; pre-018 layouts that had renamed those channels lose those names.

### Slot operations

A facility slot is either empty or bound to a channel. The user can:

- **On an empty slot:**
  - **Select channel** — opens a picker of channels of the slot's required role that are not currently bound to another slot (e.g., for a Block Indicator's input slot, the hardware-owned `block-occupancy` channels in the layout). On selection the channel is bound to the slot. Offered whenever at least one compatible channel exists.
  - **Add channel** — creates a new user-owned channel of the slot's required role, with a style chosen as part of creation (when more than one user-creatable style exists for the role), bound to a hardware target chosen as part of creation, and binds the new channel to the slot in the same action. Offered whenever at least one style for the slot's required role is user-creatable AND has at least one available hardware target. For the Block Indicator output slot, Add channel opens a sub-picker of unclaimed, constraint-compatible Direct Lamp Control rows; on selection, Bowties creates a `lamp-indicator` channel with style `single-led-direct-lamp` bound to that row, assigns it a default name, and binds it to the slot — all atomically. The user can rename it afterward.
- **On a filled slot:**
  - **Rename channel** — inline rename of the bound channel. (Hardware-owned channels can be renamed for layout meaning, but rename does not change ownership; the channel still disappears when its underlying hardware-config is cleared.)
  - **Rebind** — pick a different compatible channel via the same Select / Add UX as on an empty slot. If the previously-bound channel was user-owned and is no longer in any slot after the rebind, it is deleted.
  - **Remove from slot** — clears the slot's binding. For hardware-owned channels, the channel persists (its lifecycle is bound to the hardware-config). For user-owned channels, the channel is deleted (it can no longer exist unbound in this slice).

A channel is bound to at most one facility slot in this slice. There is no separate channel inventory surface beyond the hardware-organised Channels panel; user-owned channels exist only because they were Add-channel'd into a slot. (Future Considerations describes when a separate channel surface and channel fan-out across slots would earn their keep.)

### Live state semantics

State plumbing reuses the existing event store (spec 016). Each channel owns the event-leaf observation directly via its style: producer-side channels observe producer leaves (BOD inputs report `occupied`/`clear` as the bus reports them); consumer-side channels observe consumer leaves (lamp-indicator channels report their last commanded state as observed on the bus). A bound channel's live state is the most recently observed leaf value for its claimed pin(s); a channel whose binding has not yet produced an observation is `unknown`.

A channel with the `lamp-indicator` role and `single-led-direct-lamp` style has live state "last commanded on the bus" — there is no physical-lamp feedback from Direct Lamp Control hardware. The discoverability of this distinction is governed by FR-032 (tooltip, release notes, and user docs rather than per-row inline labels).

## Clarifications

### Session 2026-06-27

- Q: How should the facility's auto-undeploy on Wired → Incomplete (FR-022) write the consumer channel's CDI event-id fields? → A: Reuse the existing bowtie slot-detach pipeline. Spec 018 names the seam and does not specify the mechanism (no new snapshot, no new default-value rule).
- Q: Do channel names need new uniqueness or naming rules in spec 018 (default name pattern, rename validation, collision handling)? → A: No. Spec 018 only widens the channel schema (adding `role`, `style`, `ownership`). The default-name generator and rename behavior introduced in spec 015 are reused unchanged — including its lack of a uniqueness constraint — and apply to lamp-indicator channels the same way they apply to BOD channels.
- Q: What happens to a user-owned channel when its backing node is removed from the layout? → A: Out of scope for spec 018 — Bowties has no node-removal flow today. When node-removal lands, it will need to define a channel-cascade rule (analogous to BOD daughter-board clear). Tracked under Future Considerations.
- Q: Status `Live` and "auto-deploy / auto-undeploy" framing — what's the right model given Bowties' existing layered storage system and bowtie machinery? → A: Rename the completed status from `Live` to `Wired` (facility status reflects structural completeness only, not bus-sync state). Becoming Wired creates the underlying bowtie(s) using the existing bowtie creation mechanism; returning to Incomplete frees them via the existing slot-detach pipeline. Sync of those edits to disk / bus is owned by the existing layered storage system, not by spec 018. The spec describes intent (facility owns its bowtie(s)) and reuses existing seams; no new deployment, sync, or transaction machinery is introduced.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Scaffold a Block Indicator facility ahead of wiring (Priority: P1)

A layout owner who is planning a future section of their layout opens the Railroad tab and invokes **"Add facility"**. They pick the **Block Indicator** template, give it a name (e.g., "Block 5"), and confirm. The facility appears in a new Facilities section in the Railroad tab with status **Incomplete** and two empty slots: an input slot labelled per the template (e.g., "Block") and an output slot (e.g., "Indicator"). The release notes and user docs that ship with this version cover expectation-setting; the Facilities section itself carries no in-product banner or badge. Each empty slot displays "empty" and offers the slot operations described in the Slot Operations section (Select / Add as available).

The user does not yet have a BOD-8 selected on their TowerLCC nor a Signal LCC connected. Both slots stay empty. The facility persists as a scaffolded plan: 1 facility, 2 empty slots, status Incomplete. The layout file round-trips on save / close / reopen with the facility, its template identity, slot labels, and Incomplete status intact.

**Why this priority**: This story proves that facilities are first-class entities with their own lifecycle, independent of any channel binding. It ships the standalone capability to lay out the named-behavior shape of a layout ahead of having every node configured — the user can sketch their facilities and fill the slots as nodes come online. It is also the simplest exercise of the facility persistence and "Incomplete" status surfaces.

**Independent Test**: With no BOD daughter board selected and no Signal LCC connected, a user can create a Block Indicator facility, rename it, save the layout, close, and reopen — finding the facility with its name, template identity, and both empty slots exactly as left.

**Acceptance Scenarios**:

1. **Given** the user is in the Railroad tab, **When** they invoke "Add facility", **Then** they are prompted to pick a behavior template (Block Indicator is listed) and provide a name; on confirm a new facility appears with status Incomplete and one empty slot per template-declared role, within a Facilities section in the Railroad tab.
2. **Given** a facility exists with empty slots, **When** the user looks at each slot, **Then** the slot displays as empty (no channel name, no live state) and the slot's available operations reflect the current state of compatible channels and styles (e.g., the input slot may show "no channels available — select a BOD daughter board to create some" if none exist).
3. **Given** a facility exists, **When** the user invokes Rename on the facility, **Then** the facility's name updates everywhere it appears; **When** the user invokes Delete on the facility, **Then** the facility is removed (any channels bound to its slots are unbound first; user-owned channels are deleted, hardware-owned channels remain).
4. **Given** a facility with both slots empty, **When** the layout is saved, closed, and reopened, **Then** the facility, its template identity, its slot structure with their empty state, and its name round-trip exactly.

### User Story 2 — Verify hardware via the Channels panel (Priority: P1)

A layout owner has just selected a BOD daughter board on a TowerLCC connector. Before designing any layout around it, they want to confirm each input is wired correctly. They open the Railroad tab's **Channels** section and see 8 entries appear — one per configured BOD-8 input, each identified by its node and connector + input ordinal, each named with an auto-generated default (e.g., "TowerLCC-1 BOD A1"), each showing `unknown` because the bus is not yet connected, each shown as unbound. They connect to the bus; the panel updates to show the current `clear`/`occupied` state of each channel as the bus reports it. They place a wheelset on the rails of block 3; the corresponding channel row updates to `occupied` within the normal response window. They remove the wheelset; it returns to `clear`. The user has verified the BOD-8 wiring across all 8 inputs without ever creating a facility.

The same panel later shows slot/facility binding: as the user binds channels into facility slots (in US3), each bound channel is annotated with the slot/facility name; the slot's Select-channel picker lists only channels whose binding column says "unbound."

**Why this priority**: Hardware verification is a standalone workflow with real value — users tracing a wiring fault or confirming a new node want to see "is the hardware working" without committing to any facility intent. Putting this on the Channels panel also makes the rest of the model legible: the slot's Select-channel picker is just the same panel filtered to "unbound + compatible role," which is the kind of consistency that makes the channel-and-facility model click for users.

**Independent Test**: With a TowerLCC and BOD-8 configured and connected to a bus, the Channels panel shows 8 entries reflecting the live state of the BOD-8 inputs in real time, and no facility needs to exist for this to work.

**Acceptance Scenarios**:

1. **Given** a layout with a TowerLCC node and a BOD-family daughter board selected on a connector, **When** the user opens the Railroad tab, **Then** the Channels section lists 8 hardware-owned channels with role `block-occupancy` and style `bod-block-detector-input`, each identified by its node + connector + input, each with a default auto-generated name, each with live state `unknown` until the bus is connected, each shown as unbound.
2. **Given** the Channels section is showing 8 BOD-8 channels in the `unknown` state, **When** the user connects to the bus and the eager-resolution pipeline (spec 017) completes, **Then** each channel row updates to the actual `clear`/`occupied` state observed on the bus.
3. **Given** a live bus connection and the Channels section is showing live state, **When** a real BOD input transitions on the physical hardware, **Then** the corresponding channel row reflects the new state within the same response window as the existing event store delivers.
4. **Given** no facilities exist in the layout, **When** the user uses only the Channels panel to verify hardware, **Then** every BOD-8 input can be exercised and observed; the user is never prompted to create a facility for verification to function.
5. **Given** a channel is currently bound to a facility slot (per US3), **When** the user looks at its row in the Channels panel, **Then** the row shows the binding facility/slot name in the binding column; unbound channels show "unbound".
6. **Given** a channel with style `single-led-direct-lamp` exists (per US3), **When** the user looks at its row in the Channels panel, **Then** the live state column shows the most recently observed lamp-on/lamp-off command on the bus; the user can discover via tooltip on hover, the release notes, or the user docs that this is a "last commanded" indication rather than physical confirmation.

### User Story 3 — Fill the facility's slots until it becomes Wired (Priority: P1)

A layout owner who has previously scaffolded a Block Indicator facility (US1) now connects their real hardware. They select a BOD daughter board on a TowerLCC connector — 8 hardware-owned `block-occupancy` channels appear in the Channels panel (US2), all unbound.

The user returns to their scaffolded facility's input slot. The user invokes **Select channel** on it; the picker lists the unbound, compatible channels (the 8 BOD channels just created), identified by node + connector + input. The user picks one. The channel is bound to the slot; the input slot now displays the channel's name and its live state; in the Channels panel, the chosen channel's binding column now shows the facility/slot name.

The output slot is still empty. The user invokes **Add channel** on it (there are no existing compatible channels — `lamp-indicator` channels only exist if a user has previously created them in some other slot). Add channel opens the lamp-row sub-picker, listing unclaimed, constraint-compatible Direct Lamp Control rows on connected Signal LCC nodes. The user picks a row; Bowties creates a channel with role `lamp-indicator` and style `single-led-direct-lamp` bound to that row, assigns it a default name, binds it to the slot — all atomically.

The facility becomes Wired. Bowties creates the underlying bowtie(s) that realise the template's pass-through mapping using the existing bowtie creation mechanism — the same machinery a user would invoke today to create a bowtie by hand. Once those edits have reached the bus through the existing layered storage system, toggling the physical block drives the physical LED through the bus's event semantics alone — no app-side logic in the loop.

Later, the user can iterate: Rebind a slot to a different channel; Remove from slot (slot empties — for the input slot, the BOD channel returns to "unbound"; for the output slot, the user-owned LED channel is deleted and the lamp row is released, so the style's previously-locked CDI fields on that row become editable again); Delete the facility (which removes any user-owned channels bound to its slots and releases hardware-owned ones to "unbound").

**Why this priority**: This is the headline story — the architectural seams from US1 and the hardware visibility from US2 culminate in a real, physical end-to-end demonstration. It also proves that slot bindings are reversible and rebindable, which the data model gets for free from the channel-with-implementation design.

**Independent Test**: With a scaffolded Block Indicator facility from US1, a BOD daughter board selected on a TowerLCC, and a real Signal LCC connected with at least one unclaimed Direct Lamp Control row, the user selects a BOD channel for the input slot and adds an LED channel for the output slot. The facility becomes Wired; once the underlying bowtie(s) have reached the bus, physically occupying the block lights the LED; clearing the block turns it off; deleting the facility stops the behavior.

**Acceptance Scenarios**:

1. **Given** an empty input slot on a Block Indicator facility and one or more unbound BOD channels in the layout, **When** the user invokes "Select channel" on the slot, **Then** the picker lists the unbound, compatible channels by node + connector + input; on selection the chosen channel is bound to the slot; the slot displays live state from the channel; the Channels panel updates to show the slot/facility name in the channel's binding column.
2. **Given** an empty output slot on a Block Indicator facility and no existing `lamp-indicator` channels in the layout, **When** the user invokes "Add channel" on the slot, **Then** Bowties opens a sub-picker of unclaimed, constraint-compatible Direct Lamp Control rows on connected Signal LCC nodes (identified by node and row); on selection, Bowties creates a channel with role `lamp-indicator` and style `single-led-direct-lamp` bound to that row, assigns it a default name, and binds it to the slot — all as a single atomic action; the new channel appears in the Channels panel with the slot/facility name in its binding column.
3. **Given** a facility whose slots are all filled by bound channels, **When** the last slot becomes filled, **Then** the facility becomes Wired and Bowties creates the underlying bowtie(s) realising the template's pass-through mapping using the existing bowtie creation mechanism; the facility is rendered as Wired in the Railroad tab.
4. **Given** a Wired Block Indicator facility and a live bus connection, **When** the physical block transitions to occupied, **Then** the physical LED on the bound Signal LCC row turns on; **When** the block clears, **Then** the LED turns off. (End-to-end physical test.)
5. **Given** a Wired facility, **When** the user invokes "Remove from slot" on either bound channel, **Then** the facility returns to Incomplete in a single atomic action: the affected underlying bowtie(s) are removed via the existing slot-detach pipeline; once that change has reached the bus the LED stops following the block; the slot becomes empty. For the input slot, the BOD channel returns to "unbound" in the Channels panel. For the output slot, the user-owned LED channel is deleted entirely and the lamp row is released; the style's previously-locked CDI fields on the row become editable again.
6. **Given** a Wired facility, **When** the user invokes "Rebind" on the output slot and picks a different lamp row via Add channel, **Then** the previous lamp row is released (its style-locked CDI fields become editable again) and the new lamp row becomes claimed (its style-locked CDI fields lock to the style-required values); the slot's binding updates atomically; the facility remains Wired throughout the rebind. (The previously-bound LED channel, no longer in any slot, is deleted.)
7. **Given** a facility is Wired, **When** the layout is saved, closed, and reopened, **Then** the facility, all slot bindings, all channel identities, and the underlying bowtie(s) all round-trip; the facility is Wired again on reopen without any user action.
8. **Given** the new Facilities and Channels sections are visible to all users, **When** the user opens the Railroad tab, **Then** both sections are presented as plain functional content (no badges, banners, or in-product expectation-setting chrome) — expectation-setting is delivered exclusively through the release notes and user documentation that ship with this version. The Config tab gains no new badges or claim-state affordances in this feature — the only Config-tab change is that the existing profile validity/relevance mechanism now reflects the constraint contract a bound style imposes on a claimed pin's CDI fields.

### Edge Cases

- **Sub-picker filtering for Add channel on the LED output slot**: The sub-picker MUST list only Direct Lamp Control rows that are both unclaimed AND constraint-compatible with the `single-led-direct-lamp` style (i.e., rows whose `Lamp Selection` is not already "Used by Mast"). Rows that fail the compatibility check are hidden rather than offered with a confirmation step.
- **BOD daughter board cleared (or changed) while one of its channels is bound to a slot**: The hardware-owned channels for that board are deleted. Any facility slot bound to one of those channels becomes empty; the facility returns to Incomplete if it was Wired (the affected underlying bowtie(s) are removed via the existing slot-detach pipeline). The Channels panel no longer shows the deleted channels.
- **User-owned channel removed from its slot**: The slot becomes empty and the channel is deleted (no ref-counting in this slice — user-owned channels can no longer exist unbound). For LED channels, the underlying Direct Lamp Control row is released and the style's previously-locked CDI fields on that row become editable again. If the facility was Wired, it returns to Incomplete in the same atomic action.
- **Facility deletion while Wired**: Deleting the facility removes its underlying bowtie(s) via the existing slot-detach pipeline (once those edits have reached the bus the LED stops following the block). Any user-owned channels bound to its slots are deleted; any hardware-owned channels bound to its slots return to "unbound" in the Channels panel (they keep existing because the hardware-config still selects them).
- **Multiple facilities reading the same producer**: Not possible in this slice because each channel is bound to at most one slot. ABS-style producer-channel fan-out is tracked in Future Considerations.
- **Multiple facilities driving the same consumer**: Not possible in this slice because a channel is bound to at most one slot, and creating a new LED-output channel claims a previously-unclaimed lamp row.
- **Channel rename after slot binding**: The slot displays the channel's current name; renaming the channel updates the slot display immediately. No snapshotting. Hardware-owned channels can be renamed for layout meaning, but renames are lost when their underlying hardware-config is cleared (the channel itself is destroyed).
- **Drift from external CDI edits**: If the user manually edits an unmanaged Direct Lamp Control row field in the Config tab in a way that affects facility behavior, the facility's live indicator may stop tracking accurately. The managed-field constraint contract prevents the most damaging edits (e.g., flipping `Lamp Selection` back to "Used by Mast"), but does not yet detect drift originating outside Bowties (JMRI, raw CDI tools). Drift detection and one-click repair are explicitly deferred to a later feature; for this slice, repair is by Rebind or by Remove-from-slot + Add channel.
- **Connecting before discovery completes**: Bound channels inherit the spec 017 "no config data → unknown → live" indicator progression. No special handling needed; the live-state pipeline already covers this.
- **Empty layout / no facilities or channels**: The Railroad tab's Facilities section shows an empty-state message; "Add facility" remains available. The Channels panel shows whatever hardware-owned channels exist from configured hardware (zero if no BOD daughter boards are selected).
- **Consumer channel live state**: A channel with style `single-led-direct-lamp`'s row in the Channels panel shows the most recently observed lamp-on/lamp-off command on the bus — Direct Lamp Control hardware reports no physical-lamp feedback, so a row that says `lit` means "lit-event was last sent on the bus," not "the LED is physically lit." Discoverability of this distinction is governed by FR-032. Test-event support to verify consumer channels end-to-end is deferred (see Future Considerations).
- **Facility scaffolded before any compatible hardware exists**: A facility with empty slots is valid and persistable. The slot's available operations adapt: if no compatible channel exists and no compatible style supports user creation given the current hardware, Select and Add are disabled (with tooltips identifying the missing role). The slot still accepts its no-op empty state; the facility is simply Incomplete until hardware exists. Richer slot-hint UX (recommending specific hardware to add) is a vision-level concern, not a spec 018 deliverable.

## Requirements *(mandatory)*

Functional requirements are grouped by topic. Suggested implementation slicing (each slice is independently demoable) is sketched after the requirements list.

### Functional Requirements

**Channel + facility data model and persistence**

- **FR-001**: The layout MUST model **channels** as first-class, persisted entities. Each channel has a stable identity, a name, a **role** (its state vocabulary + slot-binding contract), a **style** (the specific hardware-shape realisation of the role — pins claimed, event-leaf mapping, constraint contract), a binding (the pin(s) it claims on a specific subsystem on a specific node), and an ownership flag (`hardware-owned` or `user-owned`). A channel without a role, style, or binding is not a valid persistable state in this slice; every channel is tied to specific hardware. The pre-018 channel-to-hardware-input shape MUST be retired; no parallel persistence shape is acceptable.
- **FR-002**: A role MUST declare its state vocabulary and is the binding contract a facility slot requires; a style MUST declare which role it realises. Every channel's style MUST realise the channel's role. (For this feature each role has exactly one style; the role-style registry may remain implicit. Internally these correspond to an OO-style interface/implementation pair; users see only "role" and "style".)
- **FR-003**: The layout MUST model **facilities** as first-class, persisted entities, each with a stable identity, a name, a reference to a behavior template, and an ordered list of slots. Each slot carries a slot label, a producer/consumer designation, a required channel role, and an **optional** reference to a bound channel (`channelRef` is nullable). A facility with one or more empty slots is valid and persistable.
- **FR-004**: A channel MUST be bound to at most one facility slot at a time. The system MUST enforce this invariant when binding a channel into a slot (Select channel) or when creating a new user-owned channel into a slot (Add channel).
- **FR-005**: All persisted state (channels, facilities, slot bindings) MUST round-trip fully across save, close, and reopen.
- **FR-006**: Channel ownership determines lifecycle. Hardware-owned channels (currently: BOD inputs from a selected daughter board) are auto-created when their backing hardware-config makes them necessary and auto-deleted when that hardware-config is cleared or changed; user renames do not change ownership and are lost when the channel is destroyed. User-owned channels (currently: lamp-indicator channels) are created only by explicit user action (Add channel on a facility slot) and deleted when removed from their only slot or when the facility containing their only slot is deleted; ref-counting for shared use across multiple slots is not in scope for this slice.

**Block-occupancy retrofit**

- **FR-007**: Selecting a BOD-family daughter board on a TowerLCC connector MUST atomically create one hardware-owned channel per configured input pin, each with role `block-occupancy` and style `bod-block-detector-input` bound to its pin, with a default name produced by the existing channel default-name generator from spec 015 (no new naming scheme is introduced). The producer event-leaf mapping (occupied/clear) MUST live on the style's definition, not in channel-creation code. The lifecycle of these channels MUST be bound to the daughter-board selection: clearing or changing the selection MUST atomically delete the channels (including any renamed by the user). Any facility slot bound to a deleted channel MUST become empty, and any facility that was Wired MUST return to Incomplete.
- **FR-008**: Each hardware-owned BOD channel MUST surface in the Railroad tab's Channels panel (per FR-031) with its identity (node + connector + input), its name, its live state (the most recently observed `occupied`/`clear` producer-event leaf for the claimed pin), and its binding status (the slot/facility name that has it bound, or "unbound"). The eager resolution and "no config data" indicator state from spec 017 MUST apply to each channel's live state directly — no facility needs to exist for live state to flow.
- **FR-009**: This feature ships no migration code for the pre-018 channel shape and imposes no requirement on the behavior of opening a layout saved by a pre-018 version. The spec 015 per-input channel inventory is retired by the final implementation slice; pre-018 layouts that had renamed those channels lose those names. The user is responsible for ensuring only post-018 layouts are opened.

**Lamp-indicator role and `single-led-direct-lamp` style**

- **FR-010**: The system MUST add a `lamp-indicator` role with state vocabulary `unknown`, `lit`, `unlit`.
- **FR-011**: The system MUST add a `single-led-direct-lamp` style for the `lamp-indicator` role, targeting Signal LCC's Direct Lamp Control subsystem. Each channel with this style claims exactly one Direct Lamp Control row, with consumer event-leaf mapping from `lit` to the row's lamp-on consumer event and from `unlit` to the row's lamp-off consumer event. The style MUST be marked as user-creatable (its instances come into existence via Add channel) and MUST declare a constraint contract over the row's CDI fields (see FR-029 and FR-030).
- **FR-012**: A Signal LCC node's Direct Lamp Control subsystem MUST declare in its profile that its rows are eligible to back channels with style `single-led-direct-lamp`. Eligibility makes a row available to the Add channel sub-picker (per FR-018) subject to constraint compatibility.
- **FR-013**: *(Removed.)* This feature adds no dedicated Config-tab affordances for claim-state display. Hardware claim state surfaces through (a) the Channels panel's binding column (FR-031) and (b) the natural effect of the style's constraint contract on the row's CDI fields (FR-026–FR-030) — a row whose style-managed fields are locked is, by construction, claimed.

**Behavior templates and the Block Indicator template**

- **FR-014**: The system MUST model **behavior templates** as declared entities, each with a stable template identity, an ordered list of slot definitions (each carrying a slot label, a producer/consumer designation, and a required channel role), and a mapping from producer-side semantic states to consumer-side commands.
- **FR-015**: The system MUST register a Block Indicator behavior template with two slots: input (producer, requires role `block-occupancy`) and output (consumer, requires role `lamp-indicator`), and a pass-through mapping (`occupied → lit`, `clear → unlit`).

**Facility CRUD and slot operations**

- **FR-016**: The Railroad tab MUST provide an "Add facility" action. Invoking it MUST prompt the user to pick a behavior template (Block Indicator is the only choice in this slice) and provide a name; on confirm, a new facility is created with status Incomplete and one empty slot per template-declared role.
- **FR-017**: A facility MUST be renameable and deletable from the Railroad tab. Deleting a facility MUST remove any underlying bowtie(s) it owned via the existing slot-detach pipeline. Each slot's binding MUST be cleared per the Remove-from-slot rules of FR-018 (hardware-owned channels return to "unbound" in the Channels panel; user-owned channels are deleted).
- **FR-018**: Each empty slot MUST offer the following operations as applicable:
  - **Select channel** — offered whenever at least one channel of the slot's required role exists in the layout and is not currently bound to another slot. Opens a picker of those channels; on selection, the chosen channel is bound to the slot.
  - **Add channel** — offered whenever at least one style for the slot's required role is marked user-creatable AND has at least one available hardware target (e.g., a Signal LCC is connected with at least one unclaimed, constraint-compatible Direct Lamp Control row). The action creates a new user-owned channel of the slot's required role, with a style chosen as part of the action (when more than one user-creatable style exists for the role), binds it to a hardware target chosen as part of the action, and binds the new channel to the slot — all atomically. For the lamp-indicator case, this opens a sub-picker of unclaimed, constraint-compatible Direct Lamp Control rows on connected Signal LCC nodes (identified by node and row); on selection, Bowties creates a channel with role `lamp-indicator` and style `single-led-direct-lamp` bound to that row, assigns it a default name, and binds it to the slot. Failure of any step MUST leave the layout unchanged.
- **FR-019**: Each filled slot MUST display the bound channel's name, its ownership (hardware-owned or user-owned), and its live state. The slot MUST offer:
  - **Rename channel** — inline rename of the bound channel. Permitted on both hardware-owned and user-owned channels.
  - **Rebind** — opens the same Select / Add UX as on an empty slot, scoped to compatible alternatives that are not the currently-bound channel. On selection, the slot's binding is replaced atomically. If the previously-bound channel was user-owned and is no longer in any slot after the rebind, it MUST be deleted (and its underlying hardware target, e.g., the Direct Lamp Control row, MUST return to unclaimed).
  - **Remove from slot** — clears the slot's binding. Hardware-owned channels persist (returning to "unbound" in the Channels panel) because their lifecycle is bound to their hardware-config. User-owned channels are deleted (and their underlying hardware target returns to unclaimed) because they cannot exist unbound in this slice.
  - **Rebind retired 2026-07-01 per S6 review** — the Rebind action was removed in favour of a two-step Remove-from-slot + Select/Add channel sequence, unifying the change-a-slot flow across both slot types and eliminating a shallow abstraction that duplicated the two primitives it wrapped (YAGNI). The Rename and Remove-from-slot obligations above remain in force.

**Facility lifecycle**

- **FR-020**: A facility's status MUST be derived from its slots: **Incomplete** if any slot is empty; **Wired** if all slots are filled. The Railroad tab MUST render the facility's status visibly (badge, color, or equivalent). Status reflects structural completeness only; it does not depend on whether the underlying bowtie edits have reached the bus yet.
- **FR-021**: When a facility becomes Wired (the last empty slot is filled), Bowties MUST create the underlying bowtie(s) that realise the template's producer-to-consumer mapping using the existing bowtie creation mechanism. Spec 018 introduces no facility-specific deployment path or sync logic; propagation of those edits to disk and to the bus is owned by the existing layered storage system. No separate "deploy" action is required.
  - **Composition seam pinned 2026-07-01 (S6 D2 + D6)** — the "existing mechanism" cited above is now `facilityOrchestrator.composeBowtiesIfWired`, which invokes the `compose_facility_bowties` IPC (owned by `bowties-core/src/facility_bowties/mod.rs`) and dispatches each returned `CompositionOp` through the same `configEditor.applyEdit` + `bowtieMetadataStore.createBowtie` primitives that `BowtieCatalogPanel.handleNewConnection` uses. Per D6, composition ADOPTS the producer channel's existing event IDs and writes ONLY the consumer channel's Lamp On / Lamp Off leaves — the producer's leaves are never rewritten during composition. This matches LCC's producer-identifies / consumer-subscribes contract and preserves the `resolution.writeTo === 'consumer'` invariant already honoured by the manual bowtie flow.
- **FR-022**: When a facility returns to Incomplete (slot binding removed, facility deleted, or an underlying hardware-owned channel destroyed by hardware-config change), Bowties MUST remove the affected underlying bowtie(s) via the existing bowtie slot-detach pipeline that already handles freeing a consumer slot's event-id field when a slot leaves a bowtie. Spec 018 introduces no facility-specific undeploy mechanism.
  - **Teardown + cascade pinned 2026-07-01 (S6 D3)** — `facilityOrchestrator.tearDownFacilityBowties` writes fresh event IDs onto the consumer leaves (`generateFreshEventIdForNode` + `configEditor.applyEdit`) and drops the corresponding `bowtieMetadataStore` rows in the draft layer. Hardware-owned channel loss (BOD daughter-board clear or incompatible reselection) triggers `facilityCascadeOrchestrator`, which stages detach + teardown IN THE DRAFT LAYER so cascade side effects are visible immediately next to their trigger, atomic on Save, and revertable on Discard (ADR-0012 extension). No on-save "fixup" cascades.
- **FR-023**: A Wired Block Indicator facility, once its underlying bowtie(s) have reached the bus through the existing layered storage system, MUST cause real bus behavior such that a transition of the bound input channel's underlying occupancy event drives the bound output channel's underlying lamp event accordingly, without any app-side logic mediating the bus traffic.

**Live state plumbing**

- **FR-024**: A bound channel's live state MUST reflect its style's interpretation of bus events: for channels with style `bod-block-detector-input`, the most recently observed `occupied`/`clear` producer-event leaf for the claimed pin; for channels with style `single-led-direct-lamp`, the most recently observed lamp-on/lamp-off consumer-event on the bus for the claimed row's event ids. A bound channel with no observed leaf yet is `unknown`. Discoverability of the "last commanded on bus" nature of consumer-channel state (relevant for `single-led-direct-lamp` because Direct Lamp Control reports no physical-lamp feedback) is governed by FR-032.
- **FR-025**: The existing event store, eager resolution on connect, and "no config data" indicator (spec 017) MUST apply to all bound channels (both BOD producers and LED consumers) with no special-case code paths.

**Style configuration constraints**

- **FR-026**: A style MUST be able to declare a **constraint contract** over the CDI fields of the pins its channels claim. The contract MUST be expressible as, at minimum: fixing a field to a specific value, restricting a field to a declared subset of its allowed values, and hiding a field when it would otherwise be presented. Fields not named by the contract are unmanaged and remain freely user-editable.
- **FR-027**: When a channel exists on a pin, the Config tab MUST surface the channel's style's constraint contract: managed fields appear read-only with their fixed value, with restricted pickers, or hidden, as the contract specifies. Unmanaged fields remain editable. No user action MUST be able to put a managed field into a state that would invalidate the channel's semantics.
- **FR-028**: The existing daughter-board-level field constraints for the BOD families MUST migrate to the `bod-block-detector-input` style's contract. User-visible behavior MUST be identical to today: selecting BOD-8 still applies the same restrictions to the same fields. The constraint declaration MUST live with the style, not with the daughter-board selection event.
- **FR-029**: The `single-led-direct-lamp` style MUST declare a constraint contract that, at minimum, manages `Lamp Selection` on the claimed Direct Lamp Control row so it cannot be set to a value (e.g., "Used by Mast") that would invalidate direct lamp control. Other fields on the row remain unmanaged for this feature unless a later style or template claims them.
- **FR-030**: The Add channel sub-picker for `single-led-direct-lamp` (per FR-018) MUST list only Direct Lamp Control rows whose current CDI field values are constraint-compatible with the style (e.g., `Lamp Selection` is not "Used by Mast"). Rows that fail the compatibility check are hidden rather than offered with a confirmation step.

**Channels panel**

- **FR-031**: The Railroad tab MUST include a **Channels** section that displays every channel in the layout in a hardware-organised list, grouped by node and subsystem. Each entry MUST show: the channel's ownership badge (hardware-owned or user-owned), its role and style, its identity (the pin(s) it claims), its current name, its current live state, and its binding status (the slot/facility name that has it bound, or "unbound"). The panel MUST update live as bus events arrive, using the same event store and progression that drive bound-channel live state (FR-024, FR-025). The panel MUST be functional with no facilities in the layout — hardware verification is its own use case.
- **FR-032**: Consumer-channel live state (e.g., for channels with style `single-led-direct-lamp`) represents the most recently observed command on the bus, not physical actuation. Users MUST be able to discover this distinction through at least one of: a per-row tooltip on hover, the user documentation, or the release notes. The Channels panel MUST NOT clutter every consumer-channel row with an inline "last commanded on bus" label by default — that approach was prototyped and judged too noisy for the discoverability benefit it provides. Sending test commands to exercise a consumer channel independently of any facility is deferred (see Future Considerations).

**Early-state UX indication**

- **FR-034**: The release notes for the version shipping this feature MUST describe what's new in facilities, what is likely to change, and how to provide feedback. The user documentation (under `docs/user/`) MUST include a short, dedicated section introducing facilities, calling out that it is an evolving area. Expectation-setting is delivered exclusively via release notes (FR-034) and user documentation. ### Key Entities

- **Pin**: A single piece of physical I/O on a node (one BOD input on a connector, one Direct Lamp Control row). Owned by the node's hardware configuration and addressed by subsystem + ordinal. Not directly referenced by facilities.
- **Channel**: The only first-class binding entity. A persisted record with a stable identity, a name, a **role**, a **style**, a binding (pin(s) on a subsystem on a node), and an ownership flag (`hardware-owned` or `user-owned`). Owns three things about its claimed pin(s): the semantic translation between pin-level CDI event leaves and the role's state vocabulary (e.g., `block-occupancy`, `lamp-indicator`); the live state derived from observing the relevant event leaves on the bus; and the style's constraint contract over the pins' CDI fields. Every channel is tied to specific hardware in this slice.
- **Channel role**: The user-facing label for what a channel does in the layout, paired with its state vocabulary. Examples: `block-occupancy` (states `unknown`/`occupied`/`clear`); `lamp-indicator` (states `unknown`/`lit`/`unlit`); future `signal-aspect-3-color` (states `unknown`/`stop`/`approach`/`clear`). A facility slot binds **by role** — any channel of the slot's required role can fill it, regardless of its style. State values name real-world intent rather than electrical or boolean abstractions (`true`/`false`, `on`/`off`). Every role includes `unknown` as a first-class state for "no observation yet". (Internally a role corresponds to an interface in the OO sense; the spec uses "role" everywhere user-facing.)
- **Channel style**: A declared way to realise a role on a specific subsystem/pin shape. Includes which role it realises (matching the role's state vocabulary), which subsystem it lives on, how many and which pins each instance claims, the event-leaf mapping it owns (producer or consumer side), the constraint contract it imposes on its claimed pins' CDI fields, and a marker for whether instances are user-creatable (lamp-style) or auto-created by hardware-config (BOD-style). Examples: `bod-block-detector-input` (1 input pin, realises `block-occupancy`, auto-created by BOD daughter-board selection, plus the BOD-family field constraints); `single-led-direct-lamp` (1 lamp row, realises `lamp-indicator`, user-creatable via Add channel, plus a `Lamp Selection` constraint preventing "Used by Mast"). Multiple styles may realise the same role; matching between channels and slots is by role, not by exact style name. Implicit registry in this feature; promote to explicit when a second style realises the same role. (Internally a style corresponds to an implementation class realising the role's interface; the spec uses "style" everywhere user-facing.)
- **Subsystem style catalog**: Declared in the node profile per subsystem. Lists which styles may host channels on that subsystem, and (for user-creatable styles) governs which channels the user can create on that subsystem.
- **Channel ownership**: Hardware-owned channels are created and destroyed by hardware-config choices (daughter-board selection); rename is permitted but does not change ownership and is lost when the channel is destroyed. User-owned channels are created by explicit user action (Add channel on a facility slot) and destroyed when removed from their only slot or when their containing facility is deleted; ref-counting for shared use across slots is out of scope here.
- **Behavior template**: A declared template for a facility. Names its required slots (each producer or consumer, each requiring a specific channel role) and the mapping from producer-side state to consumer-side command.
- **Facility**: A named, persisted instance of a behavior template, with one slot per template-declared slot. Each slot has an **optional** binding to a specific channel; empty slots are valid. A facility transitions automatically between Incomplete (any slot empty) and Wired (all slots filled; underlying bowtie(s) realising the template's mapping have been created using the existing bowtie creation mechanism). Owns the underlying bowtie(s) created on its behalf.
- **Facility slot**: A named position within a facility, carrying a producer/consumer designation, a required channel role, and an optional bound channel. Empty slots are valid. A channel can only be bound to one slot at a time.
- **Underlying bowtie(s)**: The same bowtie entity Bowties already supports — a producer-event-id / consumer-event-id co-configuration. A Wired facility owns one or more such bowtie(s), created using the existing bowtie creation mechanism when the facility becomes Wired and removed via the existing slot-detach pipeline when it returns to Incomplete. No new bowtie mechanism is introduced for facilities.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: In a fresh layout under the new version, selecting a BOD-family daughter board on a TowerLCC connector creates 8 hardware-owned channels with role `block-occupancy` and style `bod-block-detector-input` visible in the Railroad tab's Channels panel, with deterministic default names and binding status "unbound". On connect to the bus, each channel's row shows live `occupied`/`clear` state within the same response window as the existing event store delivers (spec 016/017 behavior preserved at the channel layer).
- **SC-002**: With no BOD daughter board selected and no Signal LCC connected, a user can create a Block Indicator facility, rename it, save the layout, close, and reopen — finding the facility with its name, template identity, and both empty slots exactly as left. The facility shows status Incomplete.
- **SC-003**: Starting from SC-001 + SC-002, the user uses the input slot's Select channel action to bind one of the hardware-owned BOD channels to the slot in a single in-app workflow; the slot displays live state from the channel and the Channels panel shows the slot/facility name on the bound channel's row. The output slot is still empty; the facility is still Incomplete.
- **SC-004**: Starting from SC-003, with a Signal LCC connected, the user uses the output slot's Add channel action → picks an unclaimed Direct Lamp Control row → a `lamp-indicator` channel is created, bound to the row, and bound to the slot atomically. The facility becomes Wired; the underlying bowtie(s) are created using the existing bowtie creation mechanism. With a real BOD-8 and a real Signal LCC connected to the bus, once those edits have reached the bus, occupying the physical block causes the bound physical LED to turn on, and clearing the block causes it to turn off (end-to-end physical observation: occupancy change → LED change in under one second).
- **SC-005**: Remove-from-slot on either bound channel returns the facility to Incomplete in a single atomic action; the affected underlying bowtie(s) are removed via the existing slot-detach pipeline; once that change reaches the bus the LED stops following the block. Re-binding (Select for the BOD slot, Add for the LED slot) returns the facility to Wired and the behavior resumes without any explicit deploy action.
- **SC-006**: A user can rebind the input slot to a different BOD channel, or the output slot to a different lamp row (via Rebind → Add on a new row), in a single workflow. The previously-bound BOD channel returns to "unbound" in the Channels panel; the previously-claimed Direct Lamp Control row is released (its style-locked CDI fields become editable again) and the previously-bound user-owned LED channel is deleted.
- **SC-007**: Clearing or changing the BOD daughter-board selection while one of its channels is bound to a facility slot atomically: deletes all hardware-owned channels for that board (including the one bound to the slot), empties any slot that was bound to a deleted channel, and returns any facility that was Wired to Incomplete. The affected underlying bowtie(s) are removed via the existing slot-detach pipeline in the same atomic step.
- **SC-008**: Deleting a Wired facility removes the underlying bowtie(s) via the existing slot-detach pipeline (once that change reaches the bus, the LED stops following the block); user-owned channels bound to its slots are deleted (lamp rows are released, so their style-locked CDI fields become editable again); hardware-owned channels bound to its slots return to "unbound" in the Channels panel (they keep existing because the daughter board still selects them).
- **SC-009**: Expectation-setting is delivered exclusively out-of-product: the version's release notes call out the feature with what's new, what may change, and how to give feedback, and the user documentation includes a short "Facilities" section marked as evolving. The Facilities and Channels sections themselves ship as plain functional content with no in-product badges, banners, or expectation-setting chrome. No new Config-tab badges or claim-state affordances ship in this feature.
- **SC-010**: Saving, closing, and reopening a layout that contains at least one facility in each lifecycle state (Incomplete with some empty slots, Wired with all slots bound) round-trips every channel, facility, slot binding, ownership flag, name, and status with no loss of identity, name, binding, or live functionality on reconnect.
- **SC-011**: With a TowerLCC and a BOD-family daughter board selected but no facilities in the layout, a user can verify each BOD-8 input by watching its row in the Channels panel: connecting to the bus updates each row to its actual state, and triggering a real BOD input on the physical hardware flips the corresponding row's state within the existing event-store response window. Verification requires no facility.

## Assumptions

- Single-user, pre-1.0 project: no production-data preservation pressure. This feature ships no migration code and imposes no requirement on pre-018 layout behavior; the user manages their own pre-018 data (FR-009).
- The existing event store, eager resolution, and indicator-state pipeline (specs 016/017) are reused unchanged. No special-case code paths are added for lamp-indicator channels or for facilities.
- Signal LCC's profile already declares Direct Lamp Control's event roles and CDI shape (consumer event ids for lamp-on / lamp-off are present and addressable). This feature extends the profile with the subsystem style catalog only; it does not redefine existing CDI semantics.
- The Config tab's existing per-field constraint UX (used today for daughter-board `validityRules` and relevance rules) is reused as the surface for implementation-class-imposed constraints. No new constraint rendering UX is built in this feature; existing read-only, restricted-picker, and hidden-field affordances are repurposed.
- "Last commanded" state for lamp-indicator channels is sufficient for this feature; physical-actual-state feedback is not available from Direct Lamp Control hardware and is not required.
- Automatic creation of the underlying bowtie(s) on the Incomplete → Wired transition (with no separate deploy action) is acceptable UX for this slice. A later feature may split off an explicit deploy step if staging changes ever becomes a need.
- Hardware-less planning beyond "facility scaffold with empty slots" requires a placeholder-node mechanism, which is Future Considerations. This slice's planning capability is intentionally limited to scaffolded facilities — channels always require real hardware to exist.
- Release notes and a short user-documentation section are sufficient expectation-setting for pre-1.0 users. No in-product chrome (banners, badges, dismissible toggles, opt-in flags, telemetry-gated rollout) is required for this slice.

## Future Considerations (out of scope)

- **YAML-defined behavior templates.** Block Indicator is a hardcoded template in this feature. The seam is shaped to accept a declarative loader later. Commit to a template DSL only when there are three or four templates of varying complexity to draw the line from real evidence.
- **Placeholder nodes — the real hardware-less planning capability.** This slice's Add facility supports scaffolding the facility shape but not pre-creating channels without hardware. The long-term answer is a placeholder-node mechanism: the user creates a placeholder Signal LCC (or BOD-8, etc.), Bowties exposes its hypothetical pins, and channels bind to those placeholder pins the same way they bind to real-node pins. When the real hardware arrives, the placeholder is "promoted" to real and bindings migrate. Placeholder nodes also enable richer planning aggregates such as "what hardware do I still need to buy?" because the planned facilities can be queried for their unsatisfied bindings.
- **Hardware planning aggregates and requirements reports.** Once placeholder nodes exist, a Planning view can answer "you need 3 more LED outputs and 2 more block detectors" by inspecting bindings to placeholder-node pins. Facilities that need richer hardware (e.g., a multi-aspect signal requiring 2 or 3 LEDs) will need user input on the desired implementation at planning time.
- **Multi-pin styles in practice.** The data model carries a list of claimed pins from day one, but the only styles in this feature claim a single pin each. The first real multi-pin style (e.g., a three-aspect signal claiming three lamp rows) lands in a later feature.
- **Constraint UX polish.** The basic enforcement of style-imposed constraints is in scope (FR-026 through FR-030). What is deferred: collapsed-by-default managed sections with "managed by [style]" annotations, drift detection from external edits (JMRI, raw CDI tools), and one-click repair workflows for fields a channel manages.
- **Railroad workspace shell.** No top-level workspace toggle, no facility-centric sidebar, no comprehension flow diagram. Facilities surface in the existing Railroad tab as a new section.
- **Behavior template library / browse UX.** This feature has exactly one template (Block Indicator). The Add facility action lists it directly. The general-purpose template-picker UI lands when there is more than one template.
- **Generalised style catalog on other Signal LCC subsystems** (Port I/O-1 etc.) and on other node families. This feature opts only Signal LCC Direct Lamp Control into the user-creatable style catalog. The mechanism is structured to generalise.
- **Pin-first / Config-tab channel creation.** This slice creates channels with style `single-led-direct-lamp` only through the slot's Add channel flow. A Config-tab entry that lets the user create a channel on a row independent of any facility (and then bind it to a slot later) is a planned future enhancement; it will require either a top-level Channels surface to manage unbound user-owned channels, or ref-counting semantics for user-owned channels.
- **Channel fan-out across multiple slots + user-owned channel ref-counting.** This slice enforces one-slot-per-channel and deletes a user-owned channel when it is removed from its only slot. The vision requires a producer channel (e.g., a `block-occupancy` channel) to feed multiple facilities at once — ABS signalling is the canonical case where one block's occupancy is read by several signal-head facilities. Relaxing one-slot-per-channel requires ref-counting (channel deletion only when its bind count reaches zero) and likely a top-level Channels-grouped-by-name surface so multi-slot channels are reasonable to manage.
- **Test events for consumer channels.** Consumer channels (e.g., those with style `single-led-direct-lamp`) only show "last commanded on bus" state because Direct Lamp Control hardware has no physical-lamp feedback. A future enhancement would let the user click a consumer channel's row in the Channels panel and send a specific state's command on the bus (e.g., "send the lit-event for this channel") for hardware verification — answering "is the wiring downstream of this channel correct" the same way the BOD verification answers it for producers.
- **Picker evolution when roles gain multiple styles.** The Add channel sub-picker today lists unclaimed Direct Lamp Control rows because there is exactly one style realising `lamp-indicator`. When additional styles realising the same role land (for example, a single LED driven from a different Signal LCC subsystem or from a different node family), the picker will evolve to group by style or to ask which to use. Styles whose hardware exposes richer state (e.g., a multi-LED signal head) realise a different domain role such as `signal-aspect-3-color` rather than `lamp-indicator`, so they appear under a different role entirely and do not affect the `lamp-indicator` picker.
- **Explicit role-style registry.** Role-to-style matching is implicit because there is exactly one style per role in this feature. Promote to an explicit registry when a second style realises the same role.
- **Conflict detection for multiple facilities driving the same consumer.** Disallowed by construction in this slice (a channel is bound to at most one slot; a Direct Lamp Control row backs at most one user-owned channel). Relaxing this for OR-style multi-facility consumers with conflict detection is a later feature.
- **Explicit deploy / undeploy action on a facility.** Today, Incomplete ↔ Wired transitions automatically create or free the underlying bowtie(s). A later feature may add an explicit deploy step (e.g., for staging changes without committing them, or for confirming wiring rewrites).
- **LCC Identify Producer / Identify Events on connect** to prime initial state of newly-bound consumer channels. Out of scope; inherits the existing behavior of spec 017.
- **Node removal and its channel-cascade rule.** Bowties currently has no flow for removing a node from a layout, so spec 018 does not define what happens to user-owned channels bound to that node. When node removal lands, it will need to specify a cascade rule analogous to the BOD daughter-board clear (delete the node's channels, empty any dependent facility slots, transition affected facilities back to Incomplete via the existing slot-detach pipeline).
