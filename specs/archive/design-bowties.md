# Bowties - Visual LCC Configuration Tool

## Overview

Bowties is a desktop application built with Tauri and Rust that makes LCC (OpenLCB) layout control accessible to hobbyists by transforming complex producer/consumer events into simple visual workflows. Users can understand their existing layout configuration at a glance and configure connections through intuitive drag-and-drop interactions — all without needing to understand Event IDs, MTIs, or CDI XML internals.

**Target Users:** Model railroad hobbyists with minimal technical background who need simple guided workflows.

**Core Value Proposition:** 
- Understand what's connected (bowtie diagrams showing "when this → do that")
- Configure elements before physical wiring
- Create event links visually by dragging elements onto bowties
- Diagnose issues with real-time event monitoring
- Single-element configuration (no scrolling through large node dialogs)

## Terminology

**Physical Connections** = Hardware wiring (button wired to Line 3 screw terminal)

**Event Links** = Logical relationships (when Button A pressed → Light B turns on)

**Elements** = CDI-defined configuration points (Line 1, Output 5, Timer 1, Event 1-6 slots)

**Devices** = User-configured physical equipment with friendly names (Front Door Button, Tower Red LED)

**Bowties** = Visual representation of event relationships showing producers on left, consumers on right, connected by shared Event ID

**Nodes** = Physical or virtual devices on the OpenLCB network

**Segments** = Major sections in CDI structure (Port I/O, Conditionals, Track Receiver)

**Groups** = Collections within segments (Lines 1-16, Timers 1-4)

**CDI (Configuration Description Information)** = XML document describing node's configuration structure (fixed per firmware version)

**Configuration Values** = Runtime data (Event IDs, names, parameters) stored in node memory

## Key LCC Protocol Characteristics

**Producer/Consumer Event Model:**
- Producers generate events (button press, sensor detection)
- Consumers respond to events (turn on light, activate relay)
- Events identified by 8-byte Event IDs (e.g., 05.02.01.02.00.00.00.03)
- Decoupled: producers don't know about consumers and vice versa
- Many-to-many: one event can have multiple producers and multiple consumers

**Typical Event ID Patterns:**
- **Producer**: Usually 1 Event ID per element (simple assignment)
- **Consumer**: Multiple Event ID slots (Event1-Event6) per element, allowing single consumer to respond to multiple different events
- **Duplicate Event IDs by design**: Common pattern — not an error condition
- **Event IDs never generated by application**: They come from nodes only, application can copy/paste between elements

**Logic Elements:**
- Timers, counters, conditionals, state machines
- No physical hardware connection
- Produce and consume events for automation

## Architecture Decisions

### Technology Stack

**Frontend:**
- Tauri (Rust backend, web frontend)
- UI Framework: TBD (React/Svelte recommended)
- Canvas library for bowties: react-flow or svelteflow
- Styling: TailwindCSS

**Backend:**
- Rust implementation of OpenLCB protocol
- TCP socket connection to GridConnect hub (port 12021 or 23)
- GridConnect ASCII frame format: `:X[HEADER]N[DATA];`
- Async I/O with tokio
- Datagram protocol for multi-frame messages

**Storage:**
- SQLite embedded database for metadata
- CDI cache on disk (per-user directory)
- Session-only storage for configuration modifications
- No cloud sync in v1

### CDI and Configuration Value Strategy

**Two Distinct Retrieval Mechanisms:**

**1. CDI XML (Schema/Structure):**
- Fixed per node type/firmware version
- Describes segments, groups, elements, data types
- Large (10-100KB), slow to retrieve (~5-30 seconds)
- **Cache Strategy:**
  - Cache key: `{manufacturer}_{model}_{software_version}.cdi.xml`
  - Location: `~/.bowties/cdi-cache/` (platform-specific app data)
  - First time seeing "Tower LCC v2.3.1" → Retrieve and cache
  - Subsequent nodes with same version → Reuse cached CDI
  - On-demand retrieval when user expands node in config view
  - Background prefetch for nodes visible in Event Bowties view
  - Manual clearing: Settings → "Clear CDI Cache" (advanced/developer use case)

**2. Configuration Values (Runtime Data):**
- Actual Event IDs, user names/descriptions, parameters
- Smaller, faster to retrieve
- **Retrieval Strategy:**
  - At startup/refresh: Event IDs + user descriptions for all elements (needed for bowtie diagrams)
  - On-demand: Other parameters (debounce, polarity, timers) when user clicks element
  - Cache in session memory
  - Track modifications with dirty flags per element
  - Write only changed values back to node (not entire CDI)

**Discovery Flow:**
1. Startup: Node discovery only (fast, <1 second) - Verify Node ID protocol
2. User expands node: Fetch/lookup CDI (one-time delay or cache hit)
3. User clicks element: Fetch configuration values for that element
4. Background: Prefetch values for nodes with event links (improves bowtie display)

## User Interface Design

### Three-View Architecture

**View 1: Configuration**
- Purpose: Setting up individual elements, configuring parameters
- UI Pattern: Miller Columns (Mac Finder style)
- Focus: Deep drilling into node structure before creating event links

**View 2: Event Bowties**
- Purpose: Understanding and creating event relationships
- UI Pattern: Canvas with bowtie diagrams
- Focus: Visual representation of "when this → do that" logic

**View 3: Event Monitor**
- Purpose: Real-time diagnostics, seeing live events, troubleshooting
- UI Pattern: Scrolling log with expandable event details
- Focus: Testing physical connections and understanding event flow

**View Switching:**
- Tab bar at top: [Configuration] [Event Bowties] [Monitor]
- Deep linking between views:
  - Click element in Bowtie → Switches to Config, pre-drilled to that element
  - Click "View Bowties" in Config → Switches to Bowties, highlights element's connections
- Keyboard shortcuts: Ctrl+1, Ctrl+2, Ctrl+3

### Configuration View - Miller Columns Pattern

```
┌──────────┬──────────┬──────────┬──────────┬──────────────────┐
│ NODES    │ SEGMENTS │ GROUPS   │ ELEMENTS │ CONFIGURATION    │
│          │          │          │          │                  │
│ Tower    │ Port I/O │ Line     │ Line 3   │ Function: Button │
│ Ctrl ▶   │   ▶      │   ▶      │   ▶      │ Active: Low      │
│          │ Condition│ Line 1   │          │ Debounce: 50ms   │
│ East     │ Track RX │ Line 2   │          │ ───────────────  │
│ Panel    │ Track TX │ Line 3   │          │ Events           │
│          │          │ ...      │          │ Event On:        │
│          │          │ Line 16  │          │ 05.02.01.02...   │
│          │          │          │          │ [Copy]           │
│          │          │          │          │                  │
│          │          │          │          │ Event Off:       │
│          │          │          │          │ 05.02.01.02...   │
│          │          │          │          │ [Copy]           │
│          │          │          │          │                  │
│          │          │          │          │ User Name:       │
│          │          │          │          │ "Front Door"     │
│          │          │          │          │                  │
│          │          │          │          │ [Revert] [Apply] │
└──────────┴──────────┴──────────┴──────────┴──────────────────┘
           ◄────────── Horizontal scroll ──────────►
```

**Column 1: Nodes (200px)**
- Hierarchical list of all discovered nodes
- Badge shows configured/total elements: "East Panel 3/16"
- Auto-refreshes when nodes added/removed
- Selection populates Column 2

**Column 2: Segments (180px)**
- Shows CDI segments for selected node
- Examples: Port I/O, Conditionals, Track Receiver, Track Transmitter
- Retrieved from cached/fetched CDI XML
- Selection populates Column 3

**Column 3: Groups (180px)**
- Shows groups within selected segment
- Examples: Lines (1-16), Timers (1-4), Outputs (1-8)
- May show replicated groups
- Selection populates Column 4

**Column 4: Elements (180px)**
- Individual configuration elements
- Status indicators:
  - ✓ Green: Fully configured
  - ⚠️ Yellow: Configured but has issues/conflicts
  - ○ Gray: Unconfigured/disabled
  - 🔧 Wrench: Unsaved changes
- For consumers with multiple event slots, shows sub-items:
  - Line 5
    - Event 1 (configured)
    - Event 2 (empty)
    - Event 3 (configured)
- Selection populates Column 5

**Column 5: Configuration Panel (flexible, 300-400px)**
- Shows editable configuration for selected element
- Fields vary by element type:
  - Function/device type
  - Type-specific parameters (debounce, polarity, delays)
  - Event ID fields with [Copy] [Paste] buttons
  - User-editable name/description
  - Location tags
- Modified fields marked with indicator (blue dot, asterisk)
- Validation messages inline
- [Revert] discards changes, [Apply] writes to node
- For consumers: List of Event 1-6 slots
  - Status indicator per slot (● configured, ○ empty)
  - Click slot to edit in-place

**Benefits of Miller Columns:**
- Shows context (where you are in 4-5 level hierarchy)
- Shows siblings at each level (all 16 lines visible)
- Efficient navigation (click to drill, backtrack visible)
- Better use of horizontal space
- Familiar pattern (Mac Finder, mobile breadcrumbs)

**Default State: Connected Only**
- Shows only bowties with both producer and consumer
- Clean, focused view
- Clear indication of unconnected count

**Expanded State: Unconnected Elements**

```
Unconnected Elements (23)  ▼

By Node → Segment

├─ East Panel (12)
│  ├─ Port I/O (8)
│  │  ├─ Line 1 / Event On →
│  │  ├─ Line 1 / Event Off →
│  │  ├─ Line 2 / Event On →
│  │  └─ Line 4 / Event On →
│  └─ Logic (4)
│     ├─ Timer 1 / Trigger →
│     └─ Timer 2 / Output →
│
└─ Tower Controller (11)
   ├─ Port I/O (9)
   │  ├─ → Line 5 / Event 1
   │  ├─ → Line 5 / Event 2
   │  ├─ → Line 6 / Event 1
   │  └─ → Line 7 / Event 3
   └─ Conditionals (2)
      └─ → Conditional 1 / Trigger
```

**Visual Indicators:**
- Producers: Left-aligned with → arrow pointing right
- Consumers: Right-aligned with → arrow pointing left
- Grouped by Node, then Segment
- Collapsible tree structure

**Bowtie Diagram Elements:**

Each bowtie shows:
- **Left side:** Producer boxes (green border)
  - Node name
  - Element path (Line 3)
  - User-friendly name (Front Door)
- **Center:** Connection with Event ID
  - Label: "When pressed", "When active", "After delay"
  - Event ID: 05.02.01.02.00.00.00.03 (truncated if needed)
- **Right side:** Consumer boxes (red border)
  - Node name
  - Element path (Line 5 / Event 3)
  - User-friendly name (Red Light)

**Multiple Producers/Consumers:**
- If event has 2 producers and 3 consumers
- Left side shows 2 boxes stacked
- Right side shows 3 boxes stacked
- All connected to same center event bubble

**Interaction Patterns:**

1. **Drag Element to Create Link:**
   - Drag producer from unconnected tree
   - Drop on existing bowtie (consumer side)
   - System copies Event ID from bowtie to dragged element
   - Element moves from unconnected to connected

2. **Drag to Existing Bowtie (Primary Workflow):**
   
   **For Producers (1 Event ID):**
   - Drag "East Panel / Line 3 / Event On"
   - Drop on bowtie with Event ID: 05.02.01.02.00.00.00.05
   - Result: Line 3 Event On = 05.02.01.02.00.00.00.05
   - Producer appears on left side of that bowtie

   **For Consumers (6 Event ID Slots) - OPTION A (Recommended):**
   - Unconnected tree shows all 6 event slots:
     ```
     └─ Tower / Line 5
        ├─ → Event 1 (empty)
        ├─ → Event 2 (configured: 05.02.01...)
        ├─ → Event 3 (empty) ← Drag this specific slot
        └─ → Event 4 (empty)
     ```
   - Drag specific slot: "Tower / Line 5 / Event 3"
   - Drop on bowtie
   - Result: Event 3 slot populated with bowtie's Event ID
   - Consumer appears on right side of bowtie with "Event 3" annotation

3. **Hover/Click Interactions:**
   - Hover bowtie → Highlight + tooltip (summary)
   - Click bowtie → Details panel or jump to config
   - Right-click → Context menu (Edit, Delete, Test, Copy Event ID)

4. **Filtering:**
   - Show All / Connected Only / Unconnected Only
   - By Node
   - By Segment type (Physical I/O, Logic)
   - Search by element name

5. **Manual Event Link Creation:**
   - [+ Manual] button → Creates empty bowtie
   - Select producer from dropdown
   - Select consumer from dropdown
   - Choose which consumer event slot (if multiple)
   - Wizard shows Event IDs for both
   - System offers to copy (which direction) or warns if both populated
   - Apply writes value

**Purpose:** Real-time diagnostic tool, see live events, troubleshoot wiring/configuration

**Layout:**

```
┌─────────────────────────────────────────────────────────────┐
│ Event Monitor                  [⏸️ Pause] [🗑️ Clear] [📋 Log]│
├─────────────────────────────────────────────────────────────┤
│                                                              │
│ 10:23:45  🟢 Event: 05.02.01.02.00.00.00.03    [Copy] [▼]  │
│           Possible producers (1):                            │
│           • East Panel / Line 3 / Event On / Front Door      │
│           Consumed by (2):                                   │
│           • Tower Ctrl / Line 5 / Event 3 → turned ON        │
│           • Hall Light / Event 1 → turned ON                 │
│                                                              │
│ 10:23:46  🟢 Event: 05.02.01.02.00.00.00.04                 │
│           Possible producers (1):                            │
│           • East Panel / Line 3 / Event Off / Front Door     │
│           Consumed by (2):                                   │
│           • Tower Ctrl / Line 5 / Event 3 → turned OFF       │
│           • Hall Light / Event 1 → turned OFF                │
│                                                              │
│ 10:23:47  🟢 Event: 05.02.01.02.00.00.00.12                 │
│           Possible producers (2):                   ⚠️       │
│           • East Panel / Line 7 / Event On                   │
│           • South Panel / Line 1 / Event On                  │
│           Consumed by (0)                                    │
│                                                              │
│ 10:23:48  🔴 Consumer Report                                 │
│           Node: Tower Ctrl / Line 5                          │
│           Listening for: 05.02.01.02.00.00.00.03             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**Event Row Components:**

1. **Timestamp:** HH:MM:SS or relative "2s ago"
2. **Event Indicator:** 
   - 🟢 Producer event (fired)
   - 🔴 Consumer report (listening)
   - ⚪ Other protocol message
3. **Event ID:** 8-byte ID, click to expand raw MTI data
4. **Actions:** [Copy] [Find in Bowties] [▼ Expand/Collapse]
5. **Possible Producers Section:**
   - Lists ALL configured producers of this Event ID
   - Full path: Node / Element / Event slot / User name
   - Warning if >1 producer (expected behavior in LCC, not error)
   - Click to jump to that element's configuration
6. **Consumed By Section:**
   - Lists all elements that responded
   - Shows action taken: "turned ON", "started timer", "activated"
   - Click to jump to that element's configuration
   - Shows which event slot matched (Event 3)

**Event Deduplication:**
- Rapid repeat events (button bounce, noisy sensor)
- Group into expandable row: "Event fired 5 times in 2 seconds [▼]"
- Expand to see individual timestamps

**Filtering:**
- Show All / Producers Only / Consumers Only / Other Protocol
- By selected node (from Bowties or Config view)
- By selected element
- By event link (show traffic for specific bowtie)
- Search by Event ID or element name

**Interaction:**
- Auto-scroll (with pause/resume)
- Click event → Highlight in Bowties view
- Drag event from monitor → Drop on bowtie to create link (future enhancement)
- Export log to file for debugging

**Use Cases:**
1. Test physical wiring: Press button, see if event appears
2. Diagnose misconfiguration: Event fires but no consumers respond
3. Identify event sources: Multiple producers, which one is this?
4. Verify linkage: Does this action trigger expected reactions?

## Core Workflows

### Workflow 1: Understanding Existing Configuration

**Scenario:** User opens layout configured by someone else, wants to understand what's set up.

1. Launch Bowties → Auto-scans network (1 second)
2. Default view: Event Bowties → Connected Only
3. See 3 bowties showing existing connections
4. Hover bowtie → Tooltip shows summary
5. Click bowtie → Jump to Configuration view, drilled to producer element
6. Miller Columns show: Node → Segment → Group → Element → Configuration
7. See full details: Event IDs, parameters, user names
8. Click "View Bowties" button → Back to Bowties view, highlights that element's connections

### Workflow 2: Configuring Element Before Physical Wiring

**Scenario:** Planning to add button to Line 3, want to configure before wiring.

1. Configuration View → Miller Columns
2. Column 1 (Nodes): Select "East Panel"
3. Column 2 (Segments): Auto-loads CDI (from cache or fetch), select "Port I/O"
4. Column 3 (Groups): Select "Line"
5. Column 4 (Elements): Select "Line 3" (shows ○ Unconfigured)
6. Column 5 (Configuration): 
   - Function type: Select "Input - Button (momentary)"
   - Active state: Select "Low"
   - Debounce: 50ms (default)
   - Event On: Shows empty field (Event ID comes from node)
   - Event Off: Shows empty field
   - User Name: Enter "Front Door Button"
   - Location: Select "East Hallway" tag
7. Click [Apply] → Writes configuration to node
8. Status changes to ✓ Configured
9. Later: Physically wire button to Line 3 terminal
10. Configuration View: Click [Test] next to element
11. Event Monitor opens, filtered to Line 3 events
12. Press button → See Event On/Off appear with Event IDs
13. Now ready to create event links

### Workflow 3: Creating Event Link (Drag to Bowtie)

**Scenario:** Front Door Button configured, want to connect to Tower Red Light.

1. Event Bowties View
2. Tower Controller already has configured outputs, producing half-bowties:
   - Existing: "→ Tower / Line 5 / Event 3 / Red Light" (unconnected consumer)
3. Expand "23 unconnected elements"
4. Find "East Panel / Line 3 / Event On / Front Door →"
5. Drag producer element
6. Drop on unconnected consumer bowtie (Event ID: 05.02.01.02.00.00.00.10)
7. System copies Event ID 05.02.01.02.00.00.00.10 to Line 3 Event On
8. Bowtie updates: Now shows both producer and consumer
9. Element moves from unconnected list to connected bowtie
10. Test: Press physical button → Event Monitor shows event → Red light turns on

### Workflow 4: Creating Event Link (Consumer with Multiple Slots)

**Scenario:** Want Tower Red Light to respond to both Front Door Button and Emergency Button.

1. Front Door → Red Light already linked (from Workflow 3)
2. Emergency Button configured on Line 8, Event On has different Event ID
3. Event Bowties View: Find bowtie "Emergency Button →" (unconnected producer)
4. Expand unconnected consumers: "Tower Controller / Port I/O"
5. See "Tower / Line 5" with sub-items:
   ```
   └─ Tower / Line 5
      ├─ → Event 1 (empty)
      ├─ → Event 2 (empty)
      ├─ → Event 3 (configured: 05.02.01...10) ← Already Front Door
      ├─ → Event 4 (empty)
      ├─ → Event 5 (empty)
      └─ → Event 6 (empty)
   ```
6. Drag "→ Event 4 (empty)"
7. Drop on "Emergency Button →" bowtie
8. System copies Emergency Button's Event ID to Line 5 Event 4
9. Now two separate bowties both target same consumer:
   - "Front Door → Tower Light (Event 3)"
   - "Emergency Button → Tower Light (Event 4)"
10. Either button now activates the same physical light

### Workflow 5: Configuring Logic Element (Timer)

**Scenario:** Want light to turn off automatically 5 minutes after button press.

1. Configuration View → Select "Tower Controller"
2. Segment: Select "Logic" or "Conditionals"
3. Group: Select "Timers"
4. Element: Select "Timer 1" (○ Unconfigured)
5. Configuration:
   - Function: "Delay Off Timer"
   - Delay: 300 seconds
   - Trigger Event: (empty, will link in Bowties view)
   - Output Event: (empty, will link in Bowties view)
   - User Name: "Light Auto-off Timer"
6. Click [Apply]
7. Switch to Event Bowties View
8. Find unconnected elements:
   - Producer: "Timer 1 / Output Event →"
   - Consumer: "→ Timer 1 / Trigger Event"
9. Drag "→ Timer 1 / Trigger" onto "Front Door / Event On" bowtie
   - Now button press triggers timer
10. Create new connection:
    - Drag "→ Tower / Line 5 / Event Off" 
    - Drop on "Timer 1 / Output →" bowtie
    - Now timer output turns off light
11. Result: Button press turns on light immediately + starts timer → 5 min later, timer turns off light

### Workflow 6: Diagnosing Issue with Event Monitor

**Scenario:** Button press doesn't turn on light, need to troubleshoot.

1. Switch to Event Monitor View
2. Press physical button
3. Monitor shows:
   ```
   🟢 Event: 05.02.01.02.00.00.00.15
      Possible producers (1):
      • East Panel / Line 3 / Event On / Front Door
      Consumed by (0)  ← Problem!
   ```
4. No consumers listening → Configuration issue
5. Click "East Panel / Line 3" link → Jump to Configuration
6. See Event On = 05.02.01.02.00.00.00.15
7. Switch to Event Bowties, find "Tower / Line 5 / Event 3"
8. Check configuration → Event 3 = 05.02.01.02.00.00.00.10 (different!)
9. Fix: Use copy/paste or drag-drop to synchronize Event IDs
10. Test again: Monitor now shows consumer responding

## Appendix: LCC Protocol Reference

**Key MTI (Message Type Indicator) Values:**
- `0x0490`: Verify Node ID (Global)
- `0x0170/0x0171`: Verified Node ID
- `0x0968`: Identify Events (Addressed)
- `0x0997`: Identify Events (Global)
- `0x04C4/04C5/04C7`: Consumer Identified (Valid/Invalid/Unknown)
- `0x0544/0545/0547`: Producer Identified (Valid/Invalid/Unknown)
- `0x19DE8`: SNIP Request
- `0x19A08`: SNIP Response (datagram)
- `0x0828`: Protocol Support Inquiry
- `0x0668`: Protocol Support Reply
- `0x20A8/20A9`: Datagram (first frame, middle, final, only)
- `0x19A28`: Datagram ACK
- `0x19A48`: Datagram NAK

**Memory Spaces:**
- `0xFF`: CDI (Configuration Description Information) - read-only
- `0xFD`: Configuration memory - read/write
- `0xFE`: All memory - read/write

**GridConnect Frame Format:**
```
:X[8 hex header]N[0-16 hex data];
Example: :X195B4123N0102030405060708;
```

**Event ID Format:**
```
8 bytes (64 bits): XX.XX.XX.XX.XX.XX.XX.XX
Display: 05.02.01.02.00.00.00.03
Internal: String "0502010200000003" or [5,2,1,2,0,0,0,3]
```
